# 工人考勤与工作系统开发文档

版本

开发类型：练手案例 

核心目标：基于 Socket.io 实现双系统实时互通，完成考勤管理、任务派发 / 反馈、内部员工聊天核心功能 

开发规范 ：前端遵循 ESLint 规则，后端代码统一缩进 / 注释，变量 / 函数采用小驼峰命名，Socket 事件采用**小写 + 下划线**命名 

## 一、项目概述

### 1.1 项目背景

项目为 Socket 通信技术练手案例，搭建 ** 工人考勤系统（管理员端）**和**工人工作系统（员工端）** 双系统，依托 Node.js+MongoDB+Vue 技术栈，通过 Socket.io 实现长连接实时通信，axios 实现非实时接口请求，满足企业基础的考勤管理、任务协同和内部沟通需求，功能设计以 “简洁落地” 为核心，不做复杂业务拓展。

### 1.2 核心开发目标

1. 考勤模块：实现员工打卡状态自动判定、管理员手动修改，支持考勤数据按条件查询 / 展示；
2. 任务模块：管理员精准派发任务，员工实时接收 / 提交任务（含报告 + 文件上传），双系统任务状态实时同步；
3. 聊天模块：支持员工按用户名搜索、添加好友，实现微信样式的一对一实时聊天，保障离线消息不丢失；
4. 基础能力：完成 JWT 身份鉴权、跨域处理、文件上传、全局异常处理，实现双系统稳定的实时通信与数据交互。

### 1.3 功能边界说明

1. 仅支持**单管理员**操作，不做多管理员权限划分、任务核实等功能；
2. 考勤模块仅做当日打卡状态管理，暂不支持月度 / 年度考勤统计报表；
3. 聊天模块仅支持员工一对一沟通，不做群聊、消息撤回 / 编辑、文件发送功能；
4. 文件上传仅支持任务报告附属文件，暂不做文件预览、在线编辑功能。

## 三、系统架构

### 3.1 整体架构

plaintext

```
┌─────────────────┐      ┌─────────────────────────────────────────┐      ┌─────────────────┐
│ 工人考勤系统    │      │ 后端服务（Node+Express）                 │      │ 工人工作系统    │
│ （管理员端）    │◄────►│ - Socket.io服务（长连接/实时通信）       │◄────►│ （员工端）      │
│ - 考勤管理      │      │ - RESTful API（非实时请求/数据操作）     │      │ - 任务接收/完成 │
│ - 任务派发      │      │ - JWT鉴权（接口/Socket统一验证）         │      │ - 打卡操作      │
│ - 数据查看      │      │ - cors跨域配置（双系统域名授权）          │      │ - 好友/聊天     │
└─────────────────┘      │ - multer文件上传（本地/云存储）          │      └─────────────────┘
                         │ - bcrypt密码加密（用户信息安全）         │
                         │ - MongoDB数据存储（Mongoose关联查询）    │
                         └─────────────────────────────────────────┘
                                    │
                                    ▼
                          ┌─────────────────────────┐
                          │ MongoDB数据库           │
                          │ - users（用户信息）     │
                          │ - attendance（考勤记录） │
                          │ - tasks（任务信息）     │
                          │ - friendRelations（好友关系） │
                          │ - chatRecords（聊天记录） │
                          └─────────────────────────┘
```

### 3.2 核心通信流程

1. **Socket 长连接**：双系统前端通过`socket.io-client`携带 JWT token 连接后端 Socket 服务，后端验证通过后建立长连接，实现**任务推送、聊天消息、状态更新**的实时互通；连接断开后前端自动重连，后端维护用户在线状态。
2. **API 接口请求**：**登录、数据查询、文件上传、好友搜索**等非实时操作，通过 axios 携带 JWT token 调用后端 RESTful API，后端统一处理并返回标准化数据。
3. **数据交互流程**：前端发起请求 / 消息→后端验证鉴权→执行业务逻辑（操作数据库 / 文件）→数据库持久化数据→后端通过 Socket 推送实时状态至对应前端→前端更新页面 / 状态。

## 四、公共基础模块设计

### 4.1 登录模块

#### 4.1.1 核心功能

- 支持管理员 / 员工按**用户名 + 密码**登录，后端验证身份后生成 JWT token；
- 前端存储 token，后续所有接口请求和 Socket 连接均携带 token 做鉴权；
- token 过期后前端自动跳转至登录页，后端统一返回未授权提示。

#### 4.1.2 业务流程

1. 前端提交`username`/`password`至后端登录 API；
2. 后端通过 bcrypt 验证密码（数据库仅存储密文），验证失败返回 400 错误；
3. 验证成功后生成 JWT token（含 userId/role/exp），返回 token 和用户基础信息；
4. 前端将 token 存储至`sessionStorage`（提升安全性，页面关闭后失效），用户信息存储至 Vuex；
5. 前端路由守卫校验 token，无 token / 过期则跳转登录页。

#### 4.1.3 核心配置

- **JWT 载荷**：`{userId: 唯一用户ID, role: admin/worker, exp: Date.now() + 2*3600*1000}`（2 小时过期）；
- **JWT 密钥**：通过后端环境变量`JWT_SECRET`配置，禁止硬编码；
- **密码加密**：bcrypt 加密轮次 10 次，注册 / 修改密码时加密，登录时对比密文。

### 4.2 全局鉴权模块

#### 4.2.1 鉴权范围

- **API 接口**：所有 /api/* 接口均需鉴权，前端在请求头携带`Authorization: Bearer {token}`；
- **Socket 连接**：连接时通过`query`携带`token={token}`，后端`connection`事件中验证，失败则拒绝连接。

#### 4.2.2 鉴权流程

1. 后端提取请求 / 连接中的 token，调用 jwt.verify 验证有效性；
2. 验证失败（过期 / 无效 / 无 token）：API 返回`code:401`，Socket 触发`disconnect`并返回错误信息；
3. 验证成功：解析出 userId/role，挂载至`req.user`（API）/`socket.user`（Socket），供后续业务逻辑使用。

### 4.3 全局返回格式

所有 API 接口返回**标准化 JSON 数据**，统一错误码定义，避免开发格式混乱：

#### 4.3.1 成功返回

json

```
{
  "code": 200,
  "msg": "操作成功",
  "data": {} // 具体业务数据，数组/对象均可
}
```

#### 4.3.2 失败返回

json

```
{
  "code": 400/401/403/404/500,
  "msg": "错误描述（如参数错误/token无效）",
  "data": null
}
```

#### 4.3.3 统一错误码

表格

| 错误码 | 含义           | 适用场景                           |
| ------ | -------------- | ---------------------------------- |
| 200    | 操作成功       | 所有正常业务处理                   |
| 400    | 参数错误       | 前端提交参数缺失 / 格式错误        |
| 401    | 未授权         | token 过期 / 无效 / 未携带         |
| 403    | 无权限         | 员工访问管理员接口 / 越权操作      |
| 404    | 资源不存在     | 查找的用户 / 任务 / 考勤记录不存在 |
| 500    | 服务器内部异常 | 后端数据库 / 文件操作失败          |

### 4.4 全局异常处理

1. **前端**：封装 axios 请求拦截器 / 响应拦截器，统一处理 401/500 等错误，通过 Element-UI 的`ElMessage`做全局提示；Socket 连接 / 重连失败也通过`ElMessage`提示用户；
2. **后端**：使用 Express 全局异常中间件，捕获所有接口 / 业务逻辑异常，统一返回`code:500`，并打印错误日志（生产环境隐藏具体错误信息）。

## 五、功能模块设计

### 5.1 工人考勤系统（管理员端）

#### 5.1.1 考勤模块

##### 核心功能

1. 考勤列表展示：按**员工 + 日期**展示打卡信息，支持分页 / 多条件筛选，包含员工基础信息、打卡状态 / 时间、累计迟到次数；
2. 打卡状态自动判定：根据员工打卡时间自动标记**缺勤（红色）/ 迟到（黄色）/ 打卡成功（绿色）**；
3. 状态手动修改：管理员有权限修改员工打卡状态，修改后同步更新数据库和累计迟到次数；
4. 数据追溯：记录考勤数据的创建 / 更新时间，支持按日期查询历史考勤。

##### 核心规则

- **正常打卡时间**：8:00-8:30 为打卡成功，8:30 之后为迟到，当日 24 点前未打卡为缺勤；
- **考勤记录粒度**：按`workerId+date`唯一存储，一个员工一天仅一条考勤记录，防止重复创建；
- **累计迟到次数**：分**总累计（lateCount）和月度累计（monthLateCount）**，仅当打卡状态为`late`时自动 + 1，管理员手动修改为`late`时同步更新；
- **修改范围**：管理员仅允许修改近 7 天的考勤状态，超过 7 天禁止修改。

##### 数据字段（attendance 集合）

表格

| 字段名         | 类型     | 说明                          | 索引       |
| -------------- | -------- | ----------------------------- | ---------- |
| _id            | ObjectId | 主键 ID                       | -          |
| workerId       | ObjectId | 员工 ID，关联 users._id       | 联合唯一   |
| date           | String   | 打卡日期，YYYY-MM-DD 格式     | 联合唯一   |
| clockInTime    | Date     | 打卡时间，无则为 null         | -          |
| clockInStatus  | String   | 打卡状态：absent/late/success | 单字段索引 |
| lateCount      | Number   | 总累计迟到次数                | -          |
| monthLateCount | Number   | 月度累计迟到次数              | -          |
| createTime     | Date     | 记录创建时间                  | -          |
| updateTime     | Date     | 记录最后更新时间              | -          |

##### 交互逻辑

1. 页面加载时，调用 API 拉取**当日**考勤数据，默认分页（每页 20 条），支持按**姓名 / 岗位 / 打卡状态 / 日期**筛选；
2. 管理员点击**修改状态**，弹出表单选择新状态，提交后调用 API 更新数据库，同步更新`updateTime`和迟到次数；
3. 数据更新后，后端通过 Socket 推送`attendance_updated`消息至对应员工端，员工端实时更新打卡状态；
4. 管理员切换查询日期时，重新调用 API 拉取对应日期的考勤数据，局部刷新列表（不整页刷新）。

#### 5.1.2 任务派发模块

##### 核心功能

1. 任务派发：通过表单选择指定员工、填写任务标题 / 内容 / 截止时间，提交后发送至对应员工端，实时触发员工端通知；
2. 任务列表展示：展示所有派发的任务，包含任务基础信息、接收员工、任务状态，支持按**员工 / 状态 / 派发日期**筛选；
3. 状态实时同步：员工提交完成任务后，管理员端任务列表实时更新状态，触发页面提示框通知；
4. 任务详情查看：点击任务可查看员工提交的完成报告和上传文件（支持下载）。

##### 核心规则

- **任务 ID 生成**：前缀 T + 时间戳 + 3 位随机数，如`T20260331100000123`，确保唯一性；
- **任务状态流转**：仅支持`pending（未完成）→ completed（已完成）`，不允许反向修改；
- **通知规则**：员工完成任务后，管理员端右上角弹出`ElMessage`提示，任务列表中对应任务状态标红提醒。

##### 数据字段（tasks 集合，与员工端共用）

表格

| 字段名        | 类型     | 说明                            | 索引       |
| ------------- | -------- | ------------------------------- | ---------- |
| _id           | ObjectId | 主键 ID                         | -          |
| taskId        | String   | 任务唯一标识                    | 单字段唯一 |
| workerId      | ObjectId | 员工 ID，关联 users._id         | 复合索引   |
| taskTitle     | String   | 任务标题（短描述）              | -          |
| taskContent   | String   | 任务详细内容                    | -          |
| deadline      | Date     | 任务截止时间                    | -          |
| createTime    | Date     | 派发时间                        | 复合索引   |
| updateTime    | Date     | 最后更新时间                    | -          |
| status        | String   | 任务状态：pending/completed     | 复合索引   |
| isRead        | Boolean  | 员工是否已读，默认 false        | -          |
| finishReport  | String   | 员工提交的完成报告，无则为 null | -          |
| uploadFileUrl | String   | 上传文件 URL，无则为 null       | -          |

##### 交互逻辑

1. 管理员点击**新建任务**，弹出表单选择员工（通过 workerId 绑定）、填写标题 / 内容 / 截止时间，提交后调用 API 创建任务；
2. 后端存储任务数据后，通过 Socket 推送`new_task`消息至对应员工端，触发员工端 alert 通知和红点提醒；
3. 管理员端任务列表默认按**创建时间倒序**展示，支持分页（每页 20 条）和多条件筛选；
4. 员工完成任务后，后端通过 Socket 推送`task_completed`消息至管理员端，触发提示框通知，同时局部刷新任务列表状态；
5. 管理员点击任务**查看详情**，可查看完成报告和文件 URL，点击 URL 可下载员工上传的文件。

### 5.2 工人工作系统（员工端）

#### 5.2.1 打卡模块

##### 核心功能

1. 打卡操作：登录后显示打卡按钮，点击完成打卡，系统自动记录打卡时间并判定状态；
2. 打卡状态展示：实时展示当日打卡状态 / 时间 / 累计迟到次数，管理员修改后实时同步更新；
3. 打卡限制：当日已打卡则隐藏打卡按钮，禁止重复打卡；打卡操作绑定当前 userId，防止代打卡。

##### 交互逻辑

1. 员工登录后，调用 API 拉取当日考勤记录，若无则显示**打卡按钮**，若有则展示打卡状态 / 时间；
2. 员工点击**打卡**，调用 API 提交打卡时间，后端自动判定状态并更新考勤记录，同步更新迟到次数；
3. 打卡成功后前端提示`ElMessage`，隐藏打卡按钮，展示打卡结果；
4. 若管理员修改打卡状态，后端通过 Socket 推送`attendance_updated`消息，前端实时更新页面展示的状态。

#### 5.2.2 任务模块

##### 核心功能

1. 任务接收：登录后拉取当日未完成任务，新任务实时触发 alert 通知，通知栏显示**未读红点**（未读任务数量）；
2. 任务查看：查看任务详情（标题 / 内容 / 截止时间），点击后标记任务为**已读**，红点数量自动减少；
3. 任务完成：填写完成报告，上传附属文件（支持格式 / 大小限制），提交后标记任务为完成，实时同步至管理员端；
4. 任务列表：展示所有当日任务，按**创建时间倒序**排列，区分未完成 / 已完成状态。

##### 核心规则

- **通知规则**：新任务推送时，若员工端处于登录状态则立即触发 alert 通知，若离线则下次登录时触发；
- **红点计数**：红点数量为**未读任务数（isRead: false）**，员工点击查看任务后，`isRead`改为 true，红点数量同步减少；
- **文件上传限制**：支持 doc/xls/pdf/zip 格式，单文件最大 100MB，前端 el-upload 做前端校验，后端 multer 做二次校验；
- **文件存储规则**：本地存储路径为`./uploads/tasks/年/月/日/`，文件命名为`taskId-原文件名-随机数`，避免重名覆盖。

##### 交互逻辑

1. 员工登录后，调用 API 拉取当日所有任务，按`isRead`统计未读数量，渲染通知栏红点；
2. 接收新任务 Socket 消息`new_task`时，触发 alert 通知，同时调用 API 刷新任务列表，更新红点数量；
3. 员工点击任务查看详情，调用 API 将`isRead`改为 true，红点数量自动减少，展示任务全量信息；
4. 员工点击**完成任务**，弹出表单填写完成报告，通过 el-upload 上传文件，提交后调用 API 更新任务状态为`completed`；
5. 后端接收后更新数据库，同时通过 Socket 推送`task_completed`消息至管理员端，员工端提示提交成功并刷新任务列表。

#### 5.2.3 好友与聊天模块

##### 核心功能

1. 好友搜索：按**username**搜索公司内其他员工（仅搜索 role:worker，排除管理员），展示关键信息；
2. 好友添加：向搜索结果发送添加请求，对方同意后加入好友列表，拒绝则收到提示；
3. 好友列表：鼠标悬浮功能栏展示好友列表，按添加时间倒序排列，支持滚动加载；
4. 实时聊天：点击好友头像进入**微信样式**聊天框，实现一对一实时消息发送 / 接收，支持历史记录分页加载；
5. 离线消息：对方离线时发送的消息自动存储，对方下次登录时实时推送，确保消息不丢失；
6. 消息已读：接收方打开聊天框后，未读消息标记为已读，发送方实时收到已读反馈。

##### 核心规则

###### 好友模块规则

- **搜索范围**：仅匹配本公司内`role:worker`且`status: true`（账号可用）的员工，隐藏电话号码等敏感信息；
- **添加限制**：同一用户间仅能发送一次添加请求，未处理前禁止重复发送；
- **请求过期**：添加请求发送后 72 小时未处理则自动失效，双方均收到过期提示；
- **好友列表刷新**：添加 / 同意 / 拒绝好友后，Socket 实时推送消息，双方好友列表自动刷新。

###### 聊天模块规则

- **会话 ID 生成**：两个 userId 按**字典序拼接**，如`user1Id_user2Id`，确保一对一聊天会话 ID 唯一；
- **消息限制**：单条消息内容最大 500 字，暂不支持表情 / 图片 / 文件发送，每秒最多发送 1 条消息防止刷屏；
- **消息展示**：微信样式，己方消息**右对齐 + 灰色气泡**，对方消息**左对齐 + 白色气泡**，消息底部显示发送时间（精确到分钟）；
- **历史记录**：默认加载最近 50 条消息，上滑触发分页加载，每次加载 50 条；
- **离线消息**：接收方 Socket 断开时（onlineStatus: false），消息存储至 chatRecords，对方登录后后端主动推送所有未读离线消息。

##### 交互逻辑

###### 好友添加流程

1. 员工在搜索框输入 username，调用 API 搜索好友，后端返回匹配的员工列表（头像 / 用户名 / 姓名 / 岗位）；
2. 员工点击**添加好友**，调用 API 发送请求，后端存储 friendRelations（status:pending）；
3. 后端通过 Socket 推送`friend_request`消息至被请求方，被请求方弹出提示框，显示 “XX 请求添加你为好友”，提供**同意 / 拒绝**按钮；
4. 被请求方操作后，后端更新 friendRelations 状态，通过 Socket 推送`friend_agreed`/`friend_rejected`消息至请求方；
5. 双方好友列表实时刷新，同意后双方均可在好友列表看到对方。

###### 实时聊天流程

1. 员工鼠标悬浮功能栏，展示好友列表，点击好友头像打开聊天框；
2. 聊天框加载时，调用 API 拉取该 chatId 的最近 50 条历史消息，按发送时间正序展示；
3. 员工输入消息点击发送，通过 Socket 发送`chat_message`消息至后端，后端存储至 chatRecords（isRead: false）；
4. 后端通过 Socket 推送`chat_message`消息至接收方，接收方聊天框实时渲染消息；若接收方离线，标记为离线消息，待其登录后推送；
5. 接收方打开聊天框后，通过 Socket 发送`message_read`消息至后端，后端将该 chatId 下的未读消息改为 isRead: true，并推送`message_read`至发送方；
6. 发送方聊天框实时显示**已读**状态，接收方上滑聊天框触发分页加载，拉取更早的历史消息。

### 6.1 集合整体设计

所有集合统一增加`createTime`/`updateTime`字段，记录数据生命周期；高频查询字段建立索引，提升查询效率；通过 Mongoose 建立集合间的关联关系，支持 populate 关联查询。

### 6.3 索引优化原则

1. **唯一索引**：用于保证数据唯一性，如 users.username、tasks.taskId、attendance (workerId,date)；
2. **单字段索引**：用于高频单条件查询，如 users.role、attendance.clockInStatus、users.onlineStatus；
3. **复合索引**：用于高频多条件查询，如 tasks (workerId, status, createTime)、chatRecords (chatId, sendTime, isRead)；
4. **索引禁用**：低查询频率、大文本字段（如 taskContent、chatContent）不建立索引，避免降低写入效率。

### 6.4 数据操作规则

1. **新增数据**：`createTime`/`updateTime`默认赋值为当前时间，由 Mongoose 中间件自动处理；
2. **更新数据**：`updateTime`由 Mongoose 中间件自动更新为当前时间，无需手动赋值；
3. **关联查询**：通过 Mongoose 的`populate`方法实现集合关联查询，如查询任务时关联 users 集合，获取员工姓名 / 头像，避免多次查询数据库；
4. **数据删除**：练手项目暂不做物理删除，核心数据（用户 / 考勤 / 任务）禁止删除，聊天记录 / 好友关系可做逻辑删除。

## 八、Socket 消息设计

### 8.1 Socket 通用规范

1. 事件名采用**小写 + 下划线**命名，如`new_task`、`chat_message`；
2. 所有消息均携带`userId`，后端通过`socket.user`解析，确保消息精准推送；
3. 重要消息（任务 / 好友 / 聊天）增加**ACK 确认机制**：前端发送消息后，后端处理完成返回`xxx_ack`消息，前端未收到则 3 秒内重发，最多重发 3 次；
4. 消息推送规则：仅推送给**指定用户**（通过 userId 筛选），禁止向所有在线用户广播，减少资源消耗。

### 8.2 Socket 连接与状态管理

1. **连接建立**：前端通过`io(SOCKET_BASE_URL, { query: { token: sessionStorage.getItem('token') } })`连接，后端验证 token 后建立连接，将`socket.user`挂载至连接实例；
2. **在线状态更新**：连接建立后，后端将 users 集合中该用户的`onlineStatus`改为`true`；连接断开（`disconnect`）后改为`false`；
3. **自动重连**：前端配置`reconnection: true, reconnectionAttempts: 5, reconnectionDelay: 3000`，断开后每 3 秒重连一次，最多 5 次，失败则提示用户；
4. **离线消息推送**：用户连接建立后，后端查询 chatRecords 中该用户为`receiverId`且`isRead: false`的消息，批量推送至前端。
